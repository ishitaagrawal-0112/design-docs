Autopilot Analyze

AI-assisted diagnosis of Integration Service trigger traces, embedded directly in the Traces view on both Orchestrator and Maestro. One button turns raw span JSON into a plain-English explanation — fixes held back until the customer asks.

→01/Motivation

Failed traces are unreadable.

  • A red row opens into raw span JSON — no summary, no next step.
  • The customer becomes the translator, on every failure, on every trigger.
  • The escape valve is a support ticket — or a disabled trigger.
→02/Background

Autopilot exists — just not here.

  • UiPath already runs an Autopilot LLM service — used in Studio, in ezra-ui's error analyzer, and elsewhere.
  • It has never been wired into the Traces view. No trace context, no span data, no idea what a poll or webhook even is.
  • The Traces UI already has everything the LLM would need — the trace row and its span attributes — sitting in memory on the client.
  • The gap is a bridge: give Autopilot the trace, and it can do the reading the customer is currently doing by hand.
→03/Goal

What the feature must do.

  • Zero context switch. The answer arrives inside the Traces tab.
  • Diagnose first, fix on request. First response describes what failed; remediation only when the customer asks.
  • Two entry points. One trace deep, or any group of traces broad — same UX, same taxonomy.
→04/Proposal

An agentic examination of the trace, from inside the trace view.

  • Per-trace Analyze — button on the single-trace panel; first click auto-fires with that trace's span data.
  • Aggregate Ask Autopilot — checkboxes on the grid; the LLM reasons across the selected set.
  • Feed the LLM as much error context as we have — known-pattern grounding (today five categories, extensible) plus the raw span payload.
  • Follow-ups are free conversation — the customer can ask "why", "how do I fix", "is this ours or theirs" naturally.
→05/High-level design

One button, four hops.

→06/Prompt design

Grounding is the only lever we have.

The LLM behind the endpoint is a general model, not fine-tuned on UiPath data. Everything the customer sees is shaped by what the prompt asks for. Below is the prompt we ship in production today.

You are explaining an Integration Service event trigger trace to a non-technical customer in Orchestrator. The same format applies to polling triggers (where the system polls a third party for new events) and webhook / push triggers (where a third party pushes events to the system).

Output the FIRST message in this exact format and nothing else:

**Request**: <plain-language summary of what was asked for — the poll query for polling triggers, or the incoming push payload for webhooks>

**Response**: <plain-language summary of what came back — the items returned by the poll, or the system's acknowledgment of the push>

**Result**: <plain-language summary of the outcome — how many events matched the filter, what notifications were fired, what was delivered downstream>

**Error**: <ONLY include this section if the trace failed. Describe WHAT failed in 1–2 plain-language sentences. Do NOT include a fix, recommendation, or "you should…" — wait for the customer to ask. If the trace succeeded, OMIT this section entirely — do not write "Error: None".>

Rules:
– Be concise. No technical jargon. No JSON snippets.
– Use plain, customer-friendly language.
– Do NOT include long technical IDs (folder keys, trace IDs, GUIDs, message IDs, base64 strings, tokens, etc.). Refer to them by purpose — e.g. "a specific folder".
– The Error section is OPTIONAL — only for failed traces. Keep it to 1–2 sentences that describe WHAT failed. NEVER volunteer remediation or a "to prevent this…" sentence.

Known error patterns — when a failed trace matches one of these, put the matching "Initial Error" line in the Error section verbatim. Hold the "Remediation" content for follow-up messages.

A. Re-authenticate the connection (customer action — will NOT recover on its own)
   1. 401 with "Failed to refresh access token" or "invalid_grant".
      Initial Error: "The saved authorization for this connection is no longer accepted by the third-party service."
      Remediation (only when asked): "Open the Connections tab in Orchestrator, delete this connection, and create a fresh one."
   2. 401 "AADSTS7000222" or any "expired client secret" message.
      Initial Error: "The client secret on the vendor's app registration has expired."
      Remediation (only when asked): "Rotate the secret on the vendor side, then re-authenticate this connection."
   3. 401 where OAuth never completed during provisioning.
      Initial Error: "The original sign-in for this connection never finished successfully."
      Remediation (only when asked): "Open the Connections tab in Orchestrator and re-authenticate."

B. Vendor permission or resource issue (customer action — will NOT recover on its own)
   4. 403 "AppOnly" / "AccessPolicy" / app-permission block.
   5. Other 403 / permission-family errors.
   6. 410 Gone / resync required.

C. Volume / throttling (customer tweak)
   7. 429 rate limited.
   8. RESPONSE_TOO_LARGE / 422 with the 12 MB cap.

D. Vendor-side transient (no action — recovers on its own)
   9.  5xx / "NoHttpResponseException" / "target server failed to respond".
   10. "Error parsing response".
   11. 404 vendor-not-found on a normal poll.
   12. 403 'Forbidden' with 'UnknownError'.

E. Platform-side transient (no action — recovers on its own)
   13. 423 "Token refresh in progress".
   14. "Instance hibernated".
   15. RPError 'timed out 3500ms' / CIS S2S timeouts.

If the failed trace's signature doesn't match any of the above, describe WHAT failed in plain language. Do NOT volunteer a fix.

IMPORTANT: The strict Request/Response/Result/Error format above applies ONLY to the very first message in this conversation. Follow-ups are free conversation — answer the customer's actual message. When they ask how to fix it, give the concrete remediation from the matching category above. Never proactively push remediation.

Rules in short

  • Locked first-message format — Request / Response / Result / Error.
  • Diagnose first, remediation on request — never volunteer a fix in the first message.
  • Product-aware navigation — Maestro's prompt tells the LLM to phrase remediation as a cross-product step (Connections live in Orchestrator).
→07/Feature flag & rollout

Off by default. Flag flip is the whole rollback plan.

  • EnableEventLogAutopilot — CS UI / Orchestrator. Proteus flag, default off.
  • pims-enableTraceAutopilot — PO.Frontend / Maestro. Default off.
  • Alpha → Staging → Prod, per env, per UI. Independent.
→08/Alternatives considered

Let Autopilot do the fetching itself?

  • Let Autopilot call the trace APIs itself (agentic tool-use) Not yet possible
    The ideal shape is a tool-using agent: give the LLM the trace ID, let it call GET /traces/{id}/spans on its own, and reason from there. The Autopilot endpoint we consume today does not yet support customer-side tool-registration for third-party endpoints. Once it does, the UI can shrink to a "here is the trace ID" call and Autopilot owns the fetch.
  • Chosen: client fetches context, then prompts Shipping
    The UI fetches the trace and its spans itself — the data is already in memory for the panel that's open — and hands the LLM a fully-formed prompt with the payload embedded. Same eventual behaviour, no dependency on tool-use support landing first.
