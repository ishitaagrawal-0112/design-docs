Autopilot Analyze

AI-assisted diagnosis of Integration Service trigger traces, embedded directly in the Traces view on both Orchestrator (ConnectionService) and Maestro (PO.Frontend). One button turns raw span JSON into a plain-English explanation — with fixes held back until the customer asks.

→01/Motivation

Customers had to translate their own trigger failures.

Event triggers are how customers wire third-party events into UiPath — "tell me when a new email lands in Outlook", "tell me when a file drops in Google Drive". Each check or push writes one row into the Traces tab. Most rows are green. When a row is red, the current experience stops there.

  • Clicking a failed trace opens a panel of raw span JSON — request/response bodies, vendor error codes, headers, tokens. No summary, no explanation, no next step.
  • The page captures what happened faithfully but never translates it. Two very different failure modes — a silent-success miss vs. an expired OAuth secret — look identical to a non-engineer skimming JSON.
  • The customer becomes the translator on every failure: read JSON → guess the pattern → search docs → open a support ticket if they're not sure. This is repeated for every trigger, every incident.
  • The same five root-cause families show up over and over (re-auth, vendor permissions, throttling, vendor blip, platform blip) — well known to on-call, invisible to the customer.
  • The result is escalations that on-call could have deflected in a sentence, and worse: customers who disable a trigger because they can't tell whether it's broken or just quiet.
Two different scenarios — a failed poll and a silent success. Both end the same way: raw JSON, no explanation. — from the mid-eval walkthrough
→02/Background

What already existed to build on.

Autopilot Analyze did not need a new backend service. Every piece it depends on was already in place, in both product shells, before this feature started.

  • The Traces tab is rendered by two different UIs against the same telemetry: ConnectionService (Angular 20, inside Orchestrator) and PO.Frontend / Maestro (React 18, standalone + micro-frontend). Both already fetch traces and spans from gallup.
  • gallup already exposes GET /triggers/{id}/traces (list) and GET /traces/{traceId}/spans (per-trace detail). No new gallup endpoints are needed.
  • UiPath already ships an Autopilot agentic-loop endpoint — POST autopilotstudio_/autopilot-everywhere/v1/tools/web-search — scoped by {org}/{tenant}. Both product shells route to it, and ezra-ui's IntegrationServiceErrorAnalyzer already uses the same endpoint.
  • The five failure categories (A–E) that customers see repeatedly are known to on-call and to the connectors team; they weren't derived here, they were catalogued from support tickets and CloudElements incident retros.
  • Maestro also carries a generic Autopilot chat surface (@uipath/apollo-react/ap-chat). Reusing it in the trace view keeps this feature visually consistent with other Autopilot experiences in the product.
→03/Goal

What the feature must do to be worth shipping.

  • Zero context switch. The explanation appears where the customer already is — inside the Traces tab — never in a support doc, a Slack thread, or another product.
  • Diagnose first, fix on request. The very first response describes what failed. Remediation is only produced when the customer explicitly asks. This is the human-in-the-loop principle for AI output: never volunteer a destructive action the customer didn't ask for.
  • Two entry points. Per-trace (one row, deep) and aggregate (any subset of the grid, patterns across polls). Same prompt shape, same taxonomy.
  • No new backend service. Ship UI-only in both codebases against endpoints that already exist. This keeps the blast radius small and the rollback path clean — a feature flag flip.
  • Ship behind a flag, default off. Per-environment enablement so we can dark-launch on Alpha, watch traffic and cost, then progress to Staging and Prod.
  • One taxonomy, two prompts. The five-category catalog lives in both prompts identically; the two prompts diverge only where the product context differs (Maestro's Connections live in Orchestrator, so its prompt tells the LLM to phrase remediation as a cross-product step).
→04/Proposal

An agentic examination of the trace, from inside the trace view.

The feature is a thin UI layer that assembles a structured prompt from the trace payload the customer is already looking at, calls the Autopilot LLM endpoint, and renders the response in a chat surface next to the trace.

  • Per-trace Analyze button lives on the Event Data section of a single trace's detail panel. First click auto-fires a prompt built from that trace's spans; subsequent messages are free-form follow-up. The customer never types the first question.
  • Aggregate Ask Autopilot button lives above the traces grid. Clicking it flips the grid into selection mode (checkbox column appears) and opens the chat pane on the right. The customer picks any subset — or "Select all on page" — and asks a question in natural language.
  • Locked first-message format. The initial response is constrained to four sections: **Request** / **Response** / **Result** / **Error**. The Error section is present only when the trace failed — never "Error: None". Follow-ups are free conversation.
  • Grounded on a five-category catalog. The system prompt embeds fifteen known patterns across five categories (A: re-auth, B: vendor permissions, C: throttling, D: vendor blip, E: internal blip). Each pattern has a paired Initial Error line (always shown) and Remediation (only when asked) line.
  • Token discipline on aggregate. Successful traces travel as one-line summaries; only non-success traces get their spans fetched and injected. "Select all" over a hundred rows still fits the context window because ~90 of them are one line each.
  • Chronic vs blip. The aggregate prompt asks the LLM to distinguish chronic (same error signature across most polls, will not self-recover) from blip (one or two transient failures in an otherwise healthy run). Chronic patterns are flagged clearly; blips are described as safe to ignore.
  • Conversation memory client-side. The endpoint is stateless. Each follow-up sends the full User: … / Assistant: … transcript back so the LLM sees continuity without a session store.
Present tense on purpose. This document describes the feature as it has been built and merged in ConnectionService on branch replay-window, and is in review on PO.Frontend branch ENGCE-58202-autopilot-analyze-traces. Everything below is code that exists, not code that will exist.
→05/High-level design

One button, four hops, no new backend.

The sequence below runs identically on both UIs. Pulses in the diagram trace the actual data path — orange for outbound requests, green for responses, grey for user actions.

Reading the diagram

  • Step 02 is conditional — the UI only fetches spans for non-success traces. Successful traces already carry the summary fields the prompt needs (eventsFetched, eventsNotified, status) directly on the row.
  • Step 04 is entirely client-side. The 5-category catalog is a string constant baked into the UI bundle, not a config the backend serves. Editing it is a UI PR.
  • Step 05's URL is anchored to window.location.origin on the CS UI side — a deliberate workaround for the MFE host stripping the leading path (see §06).
  • Steps 08–09 fire on every follow-up. No delta compression, no server session. This keeps the client trivial and the endpoint stateless.

Two entry points, one prompt shape, two frontends.

The feature ships in both Angular and React with the same UX pattern. The table below pins where each piece lives; the meaningful divergences are called out beneath it.

Where the two implementations meaningfully differ

  • Chat container. PO.Frontend uses @uipath/apollo-react/ap-chat for consistency with other Autopilot surfaces on Maestro (a mid-development refactor from a hand-rolled chat). CS UI keeps a hand-rolled Angular Material chat because Orchestrator has no Apollo React runtime.
  • Retry policy. CS UI retries the LLM call twice at 2.5 s (borrowed from TriggersService.getTriggerEventLogDetails$) — a transient 5xx does not force the customer to click again. PO.Frontend does not retry because ApChat has its own error-and-resend affordance.
  • Account/tenant resolution. CS UI runs a four-layer fallback — localStorage.cs.dev.identity → SourceInfoService → PortalShell.AccountAndTenants → JWT profile claims — because when CS runs as an MFE inside the Orchestrator host, the URL, PortalShell, and facade can each be empty at different times. PO.Frontend uses a single getApiContext() because Maestro owns its own shell.
  • MFE URL anchoring. On CS UI the endpoint URL is built with window.location.origin as an absolute prefix. Without it, HttpBaseHrefInterceptor mangles the request into orchestrator_/{account}/{tenant}/autopilotstudio_/… (double tenant prefix → 405). See ENGCE-57562.
→07/APIs & dependencies

Nothing new on the backend.

Autopilot Analyze is entirely a client-side feature. It calls two backends, both of which existed before this work:

GET /api/v1/triggers/{triggerId}/traces/{traceId}/spans

gallup — returns the span array for one trace. Called only for non-success traces on aggregate, and on-demand from the per-trace panel. Existing endpoint, unchanged.

POST /{org}/{tenant}/autopilotstudio_/autopilot-everywhere/v1/tools/web-search

Autopilot agentic-loop endpoint. Same URL the ezra-ui IntegrationServiceErrorAnalyzer uses. Auth is attached by the existing global HTTP interceptor.

Request & response shape

// Request
{
  "search_request": "<full assembled prompt — system + catalog + trace JSON + question>"
}

// Response (only the field we consume)
{
  "choices": [
    {
      "message": {
        "content": "<markdown>"
      }
    }
  ]
}

What we deliberately did not add

  • No new gallup endpoint. No POST /diagnose-trace, no server-side prompt template store.
  • No new database column, migration, or state enum.
  • No server-side conversation memory. The endpoint is stateless; the client re-sends the transcript on every turn.
  • No new authentication surface. Both the gallup call and the Autopilot POST use the same bearer token the rest of each UI already carries.
→08/Prompt design

Grounding: five failure categories, verified wording.

The system prompt is the only lever we have — the LLM behind the endpoint is a general model, not fine-tuned on UiPath data. Everything the customer sees is shaped by what the prompt asks for and what it forbids. Three rules do most of the work.

Rule 1 — Locked first-message format

The initial message must be exactly four sections. The Error section is optional and appears only when the trace failed. The LLM is explicitly told not to write Error: None.

**Request**:  <what the poll asked for, or the pushed payload>
**Response**: <what the vendor returned>
**Result**:   <how many events matched, what was notified>
**Error**:    <WHAT failed, 1–2 sentences, no fix — only on failure>

Rule 2 — Diagnose first, remediation on request

The LLM is told, at length, never to volunteer a fix in the first message. Each of the 15 known patterns has an Initial Error line (always shown) and a Remediation (only when asked) line. The LLM only emits the Remediation line when the customer's follow-up is action-oriented ("how do I fix this", "what should I do").

Cat Category Example patterns Who acts?
A Connection expired 401 invalid_grant · AADSTS7000222 expired client secret · OAuth never completed at provisioning Customer · re-auth
B Vendor permission / resource 403 AppOnly policy block · other 403 permission family · 410 Gone / resync required Customer · vendor side
C Volume / throttling 429 rate limited · RESPONSE_TOO_LARGE / 12 MB polling window cap Customer · tweak poll
D Vendor-side transient 5xx / NoHttpResponseException · Error parsing response · 404 vendor-not-found · generic 403 None · self-recovers
E Platform-side transient 423 token refresh in progress · instance hibernated · CIS S2S timeouts None · self-recovers

Rule 3 — Product-aware navigation (Maestro only)

Connections are managed in Orchestrator. On the Orchestrator (CS UI) side, that is where the customer already is — the LLM can say "open the Connections tab" without qualification. On the Maestro side, the prompt carries a CRITICAL NAVIGATION RULE that forces every remediation line to be phrased as a cross-product step: "Switch to Orchestrator (a separate UiPath product from Maestro), open the Connections tab, …".

Aggregate-only rule. The aggregate prompt adds a chronic vs. blip pattern-detection block. The LLM is asked to group failures that share a root cause ("Traces 2, 5, and 7 all failed with the same auth error") rather than describe each individually, and to reassure the customer when transient failures don't repeat across most polls.
→09/Feature flag & rollout

Default off. Independent flip per UI, per environment.

  • CS UI · Orchestrator — EnableEventLogAutopilot (Proteus feature flag). Default false. Gates both the per-trace Analyze button and the aggregate Ask Autopilot button. When the flag is off the code paths are inert — no LLM traffic can be issued.
  • PO.Frontend · Maestro — pims-enableTraceAutopilot (default flags file). Default false. Same gating scope — flipping this one flag enables both entry points.
  • Rollout order: Alpha → Staging → Ring 0 → Ring 1 → … → Prod. Both flags flip independently, so the two UIs can be enabled on different schedules.
  • Kill switch. A flag flip is the entire rollback plan. There is no cache to invalidate, no data to migrate back, no in-flight requests to drain.
  • Cost containment. Because the feature is dark by default, LLM traffic is bounded to explicitly enabled tenants. On aggregate we further cap cost by only fetching spans for non-success traces (see §04).
→10/Alternatives considered

Why not backend-side matching, why not ApChat everywhere.

  • Backend pattern matcher + static advice page Rejected
    Build a rules engine that classifies each trace by regex, and render a hard-coded advice card. Rejected — the taxonomy evolves quarterly, every new vendor pattern needs a PR, and the wording customers actually respond to isn't the wording engineers write. An LLM prompted with the catalog can generalise; a switch/case can't.
  • Dedicated POST /diagnose-trace on gallup Rejected
    Wrap the LLM call in a gallup endpoint. Rejected — it adds a service to own, deploy, and version, without adding capability. The prompt is the actual business logic and it already lives closer to the UI (the two UIs disagree on Maestro-vs-Orchestrator wording, which a shared server endpoint would flatten).
  • Auto-fire on trace open Rejected
    Analyse every failed trace the moment it opens. Rejected — customer trust in AI output degrades when the AI speaks unbidden. The human-in-the-loop principle asks for a deliberate action (a click) to invoke analysis. Also removes a foot-gun on cost.
  • Hand-rolled chat on both sides Rejected on PO.Frontend
    Ship a custom Angular / React chat surface in both codebases (as the CS UI does). Rejected on PO.Frontend mid-development — Apollo's ApChat gives us a consistent look with the rest of Autopilot on Maestro, plus a built-in Request/Close event system that saved us re-writing conversation glue. CS UI keeps its hand-rolled chat because Orchestrator has no Apollo React runtime yet.
→11/Open questions & future work

What's next.

  • Feedback loop. Add thumbs-up / thumbs-down on each assistant message. Down-votes on unmatched patterns feed a catalog-candidate queue for on-call review — the mechanism by which the 5-category catalog grows without an engineer noticing every new failure family manually.
  • Autopilot ↔ Advance Watermark integration. When the aggregate analysis detects a chronic Category A/B/C pattern (won't recover on its own), surface the Skip past this window action inline inside the chat. Diagnose → act, in one place.
  • Telemetry. Track Analyze click-through rate, ask-for-fix conversion, thumbs-down rate per category. Route through the existing product-analytics stack — no new pipeline.
  • i18n on assistant output. The LLM answers in English regardless of UI locale. Non-EN customers see EN text inside a localised UI. Options: prompt the LLM in the target locale, or add a translation pass on the response — cost vs. UX call, not yet made.
  • Prompt drift monitoring. The 5-category catalog lives in two places (Angular string, React string). Divergence is easy. Add a shared source-of-truth test that snapshots both prompts and fails CI if the catalog block diverges.
