Design document · Integration Services

Advance Watermark

A one-click way for customers to move a stuck poller past the window it keeps failing on — driven from the Traces tab, previewed before it commits, and scoped to the exact object that's stuck. This is the “solve it” half of self-serve event triggers; Autopilot Analyze is the “see it” half.

→01/Motivation

Some triggers get stuck — and today the customer can only watch.

A poller advances a marker after each successful poll — the watermark (lastPollDate) — so the next poll picks up exactly where the last one left off. That model breaks down when a window becomes un-pollable: the poller retries the same window, hits the same error, and never moves.

  • Retries alone never resolve it. The failing window is deterministic — the same request produces the same failure on every attempt, forever. Backoff and retry only change how often it fails, not whether.
  • There is no customer-facing control over the watermark. The one action that would fix it — move the marker past the bad window — is not exposed anywhere in the product.
  • The fallbacks are heavy. Re-creating the trigger loses history and re-onboards from scratch; a support ticket pulls in engineering to run a manual database update for something the customer could decide in one click.
  • Autopilot can explain it, but explanation isn't recovery. Autopilot Analyze tells the customer what is wrong; a chronic “volume / rate” failure still needs a deliberate action to get past it.
A Salesforce trigger polls for new records. Once the object crosses 100,000 rows, Salesforce returns “Exceeded 100,000 distinct IDs.” The trigger retries the same window, hits the same error — and stays stuck indefinitely. — the canonical stuck-trigger case
→02/Background

The pieces this builds on already exist.

Advance Watermark did not need a new service. The watermark, the poller state, the trace history, and the Traces UI were all already in place — this feature adds a small, safe control surface on top of them.

  • gallup (the Integration Service polling service) owns the poller state per object: lastPollDate, the poll interval, and countConsecutivePollError — the count that tells us a trigger is stuck rather than merely quiet.
  • The Traces tab already renders one row per poll (green ✓ / red ✗) in both Orchestrator (ConnectionService) and Maestro (PO.Frontend), and can expand any trace to its raw span / event data.
  • A trigger is keyed internally by an object. On a multi-object trigger, each watched object carries its own poller row and its own watermark — so “stuck” and “skip” are properties of an object, not the whole trigger.
  • Recovery today is out-of-band — a manual SQL update by on-call, or trigger re-creation. There is no API and no UI for it.
  • A separate concern: a poller in the EXCEEDED auto-suspend state is recovered by a different feature. Advance Watermark operates only on actively-scheduled triggers.
→03/Goal

Turn “watch it fail” into “skip past it,” safely.

  • One-click recovery, in place. The customer unsticks a poller from the Traces tab they're already in — no ticket, no engineering, no trigger re-creation.
  • Preview before commit. Skipping moves a watermark forward irreversibly, so the customer can always Verify first — a dry run that tests the projected window without persisting anything.
  • Show the action only when it applies. The Skip affordance appears when the object is genuinely stuck — consecutive poll errors past a threshold — not on every trigger, and never on one that has simply never polled.
  • Precise scope. A skip touches only the object that's stuck; other objects on the same trigger are untouched. It advances by exactly one poll interval per click — predictable and repeatable.
  • No new trust surface. Reuse the same authentication and folder authorization the Traces endpoint already enforces.
Non-goals. Auto-remediation without a customer click; webhook triggers (this is scoped to date-time-based pollers such as Salesforce and Outlook); and EXCEEDED auto-suspend recovery, which a separate feature owns.
→04/Proposal

Examine agentically, then act deliberately.

The customer flow is two moves on one surface: an agentic examination that reads the trace for them, and a deliberate action that moves the watermark. This document covers the action; the examination is Autopilot Analyze.

  • See it — agentic examination. When a trace fails, Autopilot diagnoses it in plain English and, for a chronic “won't self-recover” pattern, points at the skip as the remedy. Diagnose-first: the AI describes what broke; the customer decides whether to act.
  • Solve it — the Skip banner. When an object's consecutive poll errors cross the threshold, the Traces tab surfaces a Stuck-trigger banner inline. The banner reads the backend's authoritative error count — never a count of rows in a filterable grid.
  • The dialog is math-free. Opening the banner fetches a snapshot: the current watermark, the poll interval, and the projected watermark after one skip — so the UI renders “skip from X to Y” without computing anything itself.
  • Verify → Confirm.
    • Verify runs the exact skip code path as a dry run at the projected window — nothing is written — and reports whether the skip would succeed.
    • Confirm commits: the watermark moves forward by one interval and polling resumes from the new point.
  • Per-object, one interval. The action advances only the stuck object's watermark, by one interval — click again to skip further. Predictable, and isolated on multi-object triggers.
Today, all the customer can do is watch the failures pile up. With this: one click skips the trigger past the bad window, and polling resumes.
→05/High-level design

View → detect → verify → commit.

One customer session, as calls between the browser, gallup, and the poller state. Pulses trace the actual data path — orange for requests, green for responses, grey for user actions, amber for the watermark write.

Reading the diagram

  • Step 04 reads the backend, not the grid. The banner is gated on consecutivePollErrors from the poller row — a filter applied to the visible traces can never turn the banner on or off.
  • Steps 09–11 write nothing. The dry run exercises the identical advance code path against the projected window, so the preview is faithful — it just doesn't persist.
  • Step 14 is the only mutation. One UPDATE moves lastPollDate forward by one interval; polling resumes on the next scheduled poll.
→06/New APIs

A small poll-date surface on gallup.

Three trigger-scoped endpoints, added next to the existing traces endpoint. All share one auth path (see §07) and one identification contract.

Endpoint Purpose Notable contract
GET /poll-date Snapshot for the Skip dialog. Returns currentPollDate, intervalMinutes, projectedAdvancedPollDate, consecutivePollErrors.
POST /poll-date/advance Advance the watermark by one interval. dryRun folds Verify and Commit into one endpoint. dryRun:true → preview, nothing persisted. Omitted → commits the advance.
POST /poll-date/set Set the watermark to a customer-chosen time. Forward-compatible; returns 501 today, contract wired so the UI is ready.

Identification contract

POST /v3/polling/connections/{connectionId}/poll-date/advance
  • triggerId is a required UUID, validated by Joi at the edge — validation runs before auth, so a malformed request never reaches the authorizer.
  • The caller never passes an object name. gallup resolves the internal polling-row key from triggerId; the UI always has the trigger id, so the legacy object-name path was dropped from the request contract.
  • One endpoint, two modes. dryRun:true guarantees the preview runs the exact same code as the commit — the two can't drift.
→07/Auth & guardrails

Same trust model as traces; guardrails on the destructive path.

Skip moves a watermark forward irreversibly, so the safety comes from several independent layers rather than a single check.

  • Unified auth. The poll-date endpoints authenticate exactly like the traces endpoint — a UiPath Identity JWT (authenticate()) followed by folder authorization (folderAuth()) requiring Schedules.View on the trigger.
  • Source-aware. source=maestro requests use JWT-only auth (the Maestro UI); everything else (Orchestrator) also runs folder authorization. A single shared dispatcher owns both branches and the “authorizer misconfigured” 500 guard, so no handler repeats it.
  • Gated visibility. The Skip banner is shown only when consecutivePollErrors crosses the threshold — which cleanly rules out never-polled and stale-window edge cases before the customer can even reach the action.
  • Preview-before-commit. Verify (dry run) lets the customer confirm the skip would succeed without persisting; commit is a separate, explicit click.
  • Feature-flag before prod. Because the action is destructive, it ships behind a flag and rolls out per environment before general availability.
→08/Decisions

Choices already made, and why.

  • Advance by one interval, per object Chosen
    A skip moves the stuck object's watermark forward by exactly one poll interval, and touches no other object on the trigger. Predictable (the customer knows precisely what one click does), repeatable (click again to go further), and isolated (multi-object triggers stay independent). A “skip to now” jump was rejected as too blunt — it can silently discard a large, legitimately-pollable range.
  • Verify and Commit share one endpoint via dryRun Chosen
    Rather than a separate /advance/verify route, a single POST /poll-date/advance takes a dryRun flag. This is more REST-friendly (one resource, mode chosen by a flag) and — more importantly — guarantees the preview exercises the exact commit code path, so they can never diverge.
  • Gate the banner on the backend's error count, not the grid Chosen
    The Skip banner is driven by consecutivePollErrors from the poller row, never by counting failed rows in the traces grid. The grid is filterable, so a row-count gate could be turned on or off by an unrelated filter — a destructive action must read an authoritative field.
  • Resolve object name server-side; require triggerId Chosen
    The request contract requires a triggerId UUID and drops the legacy object-name path. The UI always has the trigger id, and gallup resolves the internal polling-row key from it — so there is one identification path to validate, and Joi can enforce it at the edge.
  • Exclude EXCEEDED from scope Out of scope
    Skip operates only on actively-scheduled triggers. A poller in the EXCEEDED auto-suspend state is recovered by a separate feature; folding it in here would blur two different recovery models.
→09/Open questions & future work

What's next.

  • Smart-skip recommendations. Suggest the right skip distance from the connector's known failure pattern (or the gap since last success) instead of a fixed one-interval step — so a badly-stuck trigger doesn't need many clicks.
  • Autopilot × Advance Watermark. When Autopilot diagnoses a chronic, skip-able pattern, surface the Skip action directly inside the diagnosis — diagnose and act in one place.
  • Feature-flag & rollout gate. Confirm the flag name and the per-environment rollout order before general availability, given the destructive nature of the action.
  • Poller Window Replay. The complement to advancement — re-fetch events from a past window when downstream consumers missed them. Paired with this feature, skip forward + replay back = a full recovery toolkit. Requires an idempotency model (leaning: a permanent idempotency key on the trigger record, no TTL) so replay doesn't fire duplicate downstream events.
