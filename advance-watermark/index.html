→01/Motivation

A poller can get stuck, and the customer can only watch.

A polling trigger checks a vendor on a schedule. When one check can't be completed — say the vendor refuses because there are too many records to return — that poll fails, and the trigger keeps retrying the exact same window. Same request, same failure, forever.

  • Retries can't help. The failure is deterministic — the same window fails the same way every time. Backoff changes how often it fails, never whether.
  • The trigger falls behind silently. While it's stuck on the old window, every new event lands after the frozen point and is never picked up.
  • There's no way out in the product. The customer sees a wall of red, with no action to take — the only fixes today are a support ticket or re-creating the trigger from scratch.
A Salesforce trigger polls for new records. Once the object crosses 100,000 rows, Salesforce returns “Exceeded 100,000 distinct IDs.” The trigger retries the same window, hits the same error — and stays stuck. — the canonical stuck-trigger case
→02/Background

The watermark, and why a stuck window freezes it.

The behaviour above comes straight from how pollers track progress. Understanding the watermark is enough to understand both the problem and the fix.

  • The watermark is the “last checked” point. After each successful poll a marker — lastPollDate — moves forward, and the next poll starts from there. It's what stops us re-reading the same events.
  • It only moves on success. A failed poll leaves lastPollDate where it is — so a window that can't be polled becomes a wall the watermark can never get past on its own.
  • It's per object. Each watched object on a trigger has its own poller row and its own watermark, so “stuck” is a property of one object, not the whole trigger.
  • “Stuck” is measurable. The poller row tracks consecutivePollErrors — the signal that tells us a trigger is genuinely stuck, not just quiet.
→03/Goal

Let customers move past the stuck window themselves.

  • Self-serve recovery. Give the customer a way to move the watermark past the window it's stuck on — without a ticket and without re-creating the trigger.
  • Know before you commit. Let them check whether skipping would actually work, before anything changes.
  • Safe by default. Offer it only when the trigger is genuinely stuck, and keep the customer in control of the action.
→04/Proposal

A Skip button in the Traces tab: preview, then commit.

When a poller is stuck, show the customer a button that moves the watermark one poll interval forward — and let them verify it first.

  • The Skip banner. When an object's consecutivePollErrors cross the threshold, the Traces tab shows a stuck-trigger banner with an Advance Watermark action.
  • A math-free dialog. Opening it fetches the current watermark, the interval, and the projected watermark after one skip — so the UI just shows “skip from X to Y.”
  • Verify. A dry run of the exact skip path — nothing is written — that tells the customer whether the skip would succeed.
  • Confirm. Commits the skip: the watermark moves forward one interval and polling resumes from the new point.
  • Scoped. Only the stuck object's watermark moves; one interval per click.
Today the customer can only watch the failures pile up. With this: one click moves past the bad window, and polling resumes.
→05/High-level design

End-to-end flow.

One customer session, as calls between the browser, gallup, and the poller state.

Reading the diagram

  • Step 04 reads the backend, not the grid. The banner is gated on consecutivePollErrors — a filter on the visible traces can't turn it on or off.
  • Verify writes nothing (09–11); commit is the only write (13–14). The dry run runs the same advance logic inside gallup and reports the outcome without persisting.
→06/APIs

A small poll-date surface on gallup.

Two trigger-scoped endpoints, added next to the existing traces endpoint.

Endpoint Purpose Notable contract
GET /poll-date Snapshot for the Skip dialog. Returns currentPollDate, intervalMinutes, projectedAdvancedPollDate, consecutivePollErrors.
POST /poll-date/advance Advance the watermark by one interval. dryRun:true → Verify preview, nothing persisted. Omitted → commits the advance.

Both authenticate exactly like the traces endpoint — a UiPath Identity JWT plus folder authorization (Schedules.View) — so no new trust surface is introduced.

→07/Decisions

Choices made, and why.

  • Advance by one interval, per object For now
    One click moves the stuck object's watermark forward by exactly one poll interval, and touches nothing else. It's predictable and repeatable (click again to go further). Letting the customer pick a custom target date — “skip to here” — is the natural next step, wired behind POST /poll-date/set for later; a blunt “skip to now” was avoided because it can silently discard a legitimately-pollable range.
  • Verify and Commit share one endpoint via dryRun Chosen
    Rather than a separate verify route, POST /poll-date/advance takes a dryRun flag. This guarantees the preview runs the exact commit code path — so what Verify shows can never drift from what Confirm does.
  • Gate the banner on the backend's error count Chosen
    The banner is driven by consecutivePollErrors on the poller row, never by counting failed rows in the traces grid. The grid is filterable, so a row-count gate could be flipped on or off by an unrelated filter — a destructive action must read an authoritative field.
→08/Feature flag & rollout

Off by default. Flag flip is the whole rollback plan.

  • Destructive → gated. Because a skip moves the watermark irreversibly, the action ships behind a feature flag, default off, added before production.
  • Per UI. gallup owns the endpoint; each Traces UI (Orchestrator / Maestro) gates its own Skip affordance, so the two can enable independently.
  • Rollout. Alpha → Staging → Prod, per environment — dark-launch, watch, then progress.
  • Kill switch. Flipping the flag off is the entire rollback — no data to migrate back.
